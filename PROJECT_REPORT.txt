1. PROJECT OVERVIEW
================================================================================

DiabetesRisk AI (also branded as "HealthGuard AI") is an end-to-end
machine-learning-powered health risk prediction platform built with a full
MLOps workflow. The project predicts the risk of Type-2 diabetes and
cardiovascular (heart) disease using clinical health indicators. It combines
classical ML classification models, real-time SHAP explainability, LLM-powered
medical assistants (Google Gemini and OpenRouter/GPT-4o-mini), a React-based
clinical dashboard, FastAPI backend, MLflow experiment tracking, Docker-based
deployment, and GitHub Actions CI/CD automation.

The platform is designed for educational, research, and demonstration purposes
to showcase how modern MLOps practices can be applied to healthcare prediction.


================================================================================
2. PROBLEM STATEMENT
================================================================================

Diabetes and cardiovascular disease are among the leading causes of death and
disability worldwide. Early identification of at-risk individuals can
dramatically improve health outcomes through timely intervention. However:

  - Traditional screening processes are slow, expensive, and not scalable.
  - Machine learning models are often "black boxes" that clinicians cannot trust.
  - Most ML projects stop at the training stage and never reach production.
  - Model performance degrades over time as population data drifts.

This project addresses ALL of these challenges by building a complete system
that:

  1. Predicts diabetes and heart disease risk from easily obtainable health
     indicators (no blood tests required for basic screening).
  2. Explains every prediction using SHAP values so clinicians can understand
     WHY a patient is flagged as high-risk.
  3. Provides a production-ready deployment with Docker, CI/CD, and monitoring.
  4. Detects data drift and supports automated model retraining.
  5. Integrates LLMs for AI-powered health guidance and symptom triage.


================================================================================
3. PROJECT OBJECTIVES
================================================================================

  - Build a multi-model ML pipeline that trains, evaluates, and selects the
    best classifier for diabetes and heart disease prediction.
  - Provide transparent, explainable AI using SHAP (Shapley Additive
    Explanations) for every prediction.
  - Create an interactive, clinical-grade web dashboard for health professionals
    and users to input health parameters and receive risk assessments.
  - Implement full MLOps: experiment tracking (MLflow), containerisation
    (Docker), CI/CD (GitHub Actions), and data drift detection.
  - Integrate LLMs (Google Gemini 3.5 Flash and OpenRouter GPT-4o-mini) for
    AI-powered clinical narratives, symptom analysis, and chatbot assistance.
  - Store prediction history in a database for longitudinal patient tracking.
  - Support user authentication (email/password and Google OAuth via Firebase).

4. DATASETS USED
================================================================================

4.1 DIABETES DATASET
---------------------
  Name:        BRFSS 2015 (Behavioral Risk Factor Surveillance System)
  File:        data/diabetes_012_health_indicators_BRFSS2015.csv
  Source:      CDC / Kaggle
  Rows:        253,680
  Columns:     22 (21 health indicators + 1 target)
  Target:      Diabetes_012 (0 = No Diabetes, 1 = Pre-diabetes, 2 = Diabetes)
               → Binarised to: 0 = No Diabetes, 1 = Diabetes/Pre-diabetes
  File Size:   ~22 MB

  Features (21 health indicators):
  - HighBP, HighChol, CholCheck, BMI, Smoker, Stroke,
    HeartDiseaseorAttack, PhysActivity, Fruits, Veggies,
    HvyAlcoholConsump, AnyHealthcare, NoDocbcCost, GenHlth,
    MentHlth, PhysHlth, DiffWalk, Sex, Age, Education, Income

  Note: The "Income" feature is DROPPED during training to avoid
  socioeconomic bias in predictions.

4.2 HEART DISEASE DATASET
--------------------------
  Name:        Cardiovascular Disease Dataset (cardio_train.csv)
  File:        data/cardio_train.csv
  Source:      Kaggle
  Rows:        ~70,000
  Format:      Semicolon-separated (;)
  Target:      cardio (0 = No Heart Disease, 1 = Heart Disease)
  File Size:   ~2.9 MB

  Original Features:
  - id, age (in days), gender, height, weight, ap_hi (systolic BP),
    ap_lo (diastolic BP), cholesterol, gluc, smoke, alco, active, cardio

  Engineered Features:
  - age_years = age / 365.25 (converted from days to years)
  - bmi = weight / (height/100)^2 (computed Body Mass Index)
  - gender = gender - 1 (converted to 0/1 encoding)


================================================================================
5. DATA PREPROCESSING & FEATURE ENGINEERING
================================================================================

5.1 DIABETES PIPELINE
---------------------

  Step 1: Load CSV with pandas
  Step 2: Handle missing values using median imputation
  Step 3: Drop "Income" column to avoid socioeconomic bias
  Step 4: Binarise target: Diabetes_binary = (Diabetes_012 > 0).astype(int)
  Step 5: Stratified 70/15/15 train/validation/test split
  Step 6: SMOTE (Synthetic Minority Oversampling Technique)
           - Balances the imbalanced classes (majority class ~86% non-diabetic)
           - Applied ONLY on the training set (not validation/test)
  Step 7: SelectKBest(chi2, k=12) — Feature selection
           - Reduces 20 features to the top 12 most statistically significant
           - Uses chi-squared test for feature ranking
  Step 8: Save selected feature names to models/feature_names.json
  Step 9: Compute training statistics (mean, std, min, max) per feature
           for drift detection → saved to models/training_stats.json

  Selected 12 Features (Diabetes):
    HighBP, HighChol, BMI, Stroke, HeartDiseaseorAttack, PhysActivity,
    HvyAlcoholConsump, GenHlth, MentHlth, PhysHlth, DiffWalk, Age

5.2 HEART DISEASE PIPELINE
---------------------------

  Step 1: Load CSV (semicolon separator)
  Step 2: Handle missing values using median imputation
  Step 3: Drop duplicate rows
  Step 4: Feature engineering:
           - age_years = age / 365.25
           - bmi = weight / (height/100)^2
           - gender = gender - 1
  Step 5: Select 10 features for training
  Step 6: Stratified 70/15/15 split
  Step 7: SMOTE oversampling on training set
  Step 8: SelectKBest(f_classif, k=10) — uses ANOVA F-test
  Step 9: Save training stats for drift detection

  Selected 10 Features (Heart):
    gender, age_years, bmi, ap_hi, ap_lo, cholesterol, gluc,
    smoke, alco, active

6. MACHINE LEARNING MODELS USED
================================================================================

Both pipelines train and compare the following 5 classifiers. Each model is
wrapped in a scikit-learn Pipeline with StandardScaler for consistent
preprocessing during training and inference.

  ┌──────────────────────────┬─────────────────────────────────────────────┐
  │ Model                    │ Key Hyperparameters                        │
  ├──────────────────────────┼─────────────────────────────────────────────┤
  │ Logistic Regression      │ C=1.0, max_iter=1000                       │
  │ Decision Tree            │ max_depth=10                               │
  │ Random Forest            │ n_estimators=200, max_depth=12, n_jobs=-1  │
  │ Support Vector Machine   │ kernel=rbf, C=1.0, probability=True        │
  │ XGBoost                  │ n_estimators=300, max_depth=6, lr=0.1      │
  └──────────────────────────┴─────────────────────────────────────────────┘

  Additional notes:
  - SVM is subsampled to 30,000 rows (O(n^2) computational cost).
  - All models use random_state=42 for reproducibility.
  - StandardScaler normalises features before classification.
  - Each model is wrapped in: Pipeline([("scaler", StandardScaler()), ("clf", ...)])

  The older version of the training script (model_summary.json) also included
  Gradient Boosting (sklearn GradientBoostingClassifier) in addition to the
  above models.

7.3 MODEL SELECTION CRITERION
-------------------------------
  The best model is selected by the highest MACRO F1 SCORE on the test set.
  F1 is chosen over accuracy because the datasets are imbalanced (especially
  the diabetes dataset where ~86% of samples are non-diabetic). F1 balances
  precision and recall, making it more suitable for medical classification.

9. EXPLAINABILITY (SHAP)
================================================================================

SHAP (SHapley Additive exPlanations) is integrated for transparent, explainable
AI predictions:

  - TreeExplainer is used for tree-based models (Random Forest, XGBoost,
    Gradient Boosting, Decision Tree).
  - LinearExplainer is used for linear models (Logistic Regression).
  - Per-prediction SHAP values are computed for every API call.
  - Top 5 most influential features are returned with:
      * Feature name
      * Impact magnitude
      * Direction ("increases" or "decreases" risk)
  - A human-readable SHAP summary narrative is generated.
  - A global SHAP feature importance plot is saved as models/shap_global.png
    during training.
  - If SHAP computation fails, a scaled-input heuristic fallback is used.

  Example SHAP output:
    "High risk — primarily driven by: High BP, BMI, General Health."

       → Multi-turn conversation about SHAP values, clinical parameters,
         MLOps concepts, and health guidance.
  - Fallback:    High-fidelity simulated responses when GEMINI_API_KEY is not

10.2 OPENROUTER / GPT-4o-mini (Backend API)
---------------------------------------------
  - API:         OpenRouter (openrouter.ai/api/v1)
  - Model:       openai/gpt-4o-mini (default, configurable)
  - Used for:
    a) Chat Endpoint (/chat)

    b) Personalised Recommendations
       → Generates 3-6 tailored, actionable health recommendations based on
         user's health indicators and prediction results.
  - Config:      OPENROUTER_API_KEY, OPENROUTER_MODEL environment variables.

10.3 OLLAMA / Llama 3.1 (Backend API — Optional)
--------------------------------------------------
  - URL:         http://localhost:11434 (local Ollama instance)
  - Model:       llama3.1
  - Used for:    Generating personalised health recommendations when available.
  - Timeout:     6 seconds (fast fallback if unavailable).

    1. Try Ollama (local LLM) → if available, return AI-generated recommendations

       (heuristic rules based on BMI, blood pressure, cholesterol, etc.)

11. SYSTEM ARCHITECTURE
================================================================================

  The runtime architecture follows a clean three-tier design:

  ┌─────────────────────────────────────────────────────────────────────────┐
  │                         FRONTEND (React + Vite)                       │
  │  Port 3000 | Express server + Vite middleware                         │
  │  Clinical Dashboard | Predictor Forms | SHAP Visualizations           │
  │  Chatbot | Drift Console | Symptom Analyzer                          │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                             ↓  HTTP  ↓                                │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                     BACKEND API (FastAPI + Uvicorn)                    │
  │  Port 8000 | ML Inference | SHAP Computation | Drift Detection        │
  │  User Auth | Prediction History | Alerts | OpenRouter Chat            │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                             ↓  SQLite  ↓                              │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                     DATABASE (SQLite)                                  │
  │  Tables: users, predictions, alert_settings, alerts                   │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                     MLFLOW (Experiment Tracking)                       │
  │  Port 5000 | Parameters | Metrics | Model Registry                   │
  └─────────────────────────────────────────────────────────────────────────┘

  FLOW:
    1. User opens the React dashboard and enters health parameters.
    2. Frontend sends a POST request to FastAPI (/predict or /predict-heart).
    3. FastAPI loads the saved model pipeline from disk (best_model_v2.pkl).
    4. The model returns prediction, probabilities, and SHAP explanations.
    5. Results are displayed as a clinical-style risk assessment.
    6. Prediction is logged to SQLite for history tracking.
    7. Optionally, Gemini generates extended clinical narratives.

13. BACKEND API (FastAPI)
================================================================================

The FastAPI backend (api/main.py, v3.0.0) exposes the following endpoints:

  SYSTEM ENDPOINTS:
  ─────────────────
  GET  /health              → System health check (model loaded, SHAP ready,
                               drift stats available, heart model status)
  GET  /model-info          → Returns the full model_summary.json
  GET  /model-metrics       → Returns all model metrics + best model + features
  GET  /feature-importance  → Returns global feature importance values and
                               optional SHAP plot path

  PREDICTION ENDPOINTS:
  ─────────────────────
  POST /predict             → Diabetes risk prediction
                               Input: 13 BRFSS health indicators
                               Output: prediction, label, probabilities,
                               confidence %, risk level, SHAP explanation,
                               recommendations, drift detection, alerts

  POST /predict-heart       → Heart disease risk prediction
                               Input: 10 cardiovascular features
                               Output: same schema as /predict with
                               heart-specific labels

  POST /predict-symptoms    → Symptom-based diabetes prediction
                               Input: symptom list, severity, age, BMI
                               Output: derived clinical inputs + prediction

  MONITORING ENDPOINTS:
  ─────────────────────
  POST /log-input           → Log raw input for drift tracking (no prediction)
  GET  /user-history        → Retrieve user's prediction history (paginated)

  AUTHENTICATION ENDPOINTS:
  ─────────────────────────
  POST /auth/register       → Email/password registration
  POST /auth/login          → Email/password login
  POST /auth/google         → Google OAuth login (via Firebase token)

  ALERT ENDPOINTS:
  ────────────────
  POST /alert-config        → Set per-user risk alert threshold and channel
  GET  /alerts              → Retrieve user's alert history

  CHATBOT ENDPOINT:
  ─────────────────
  POST /chat                → Health guidance chatbot via OpenRouter/GPT-4o-mini

  RISK LEVEL MAPPING:
  ───────────────────
  - Low:       probability < 25%
  - Moderate:  25% ≤ probability < 50%
  - High:      50% ≤ probability < 75%
  - Very High: probability ≥ 75%

18. CI/CD (GitHub Actions)
================================================================================

File: .github/workflows/ci-cd.yml

The CI/CD pipeline runs on push to main/develop branches and consists of
7 stages:

  STAGE 1: CHECKOUT & SETUP
  ─────────────────────────
  - Checkout code
  - Setup Python 3.11
  - Cache pip dependencies
  - Install requirements.txt

  STAGE 2: LINT
  ─────────────
  - Ruff lint check on ml/ and api/ directories
  - mypy type-check on api/main.py

20. DATA DRIFT DETECTION
================================================================================

The system implements statistical data drift detection using z-score analysis:

  MECHANISM:
  ──────────
  1. During training, per-feature statistics (mean, std, min, max) are
     computed on the ORIGINAL training data (pre-SMOTE) and saved to
     models/training_stats.json.

  2. At inference time, each incoming prediction request's feature values are
     compared against the training distribution.

  3. For each feature: z = |x_input - mean| / std
     If z > 3.5 (more than 3.5 standard deviations from training mean),
     the feature is flagged as anomalous.

  4. If 2 or more features are anomalous → DRIFT DETECTED.

  5. Drift status is included in every prediction response.

  FRONTEND DRIFT CONSOLE:
  ───────────────────────
  - Displays Kolmogorov-Smirnov drift statistics
  - Shows current vs. baseline epoch comparison
  - Provides a "Retrain" button that simulates an automated retraining
    pipeline with live terminal-style logs
  - Displays post-retrain improvement metrics

21. ALERT SYSTEM
================================================================================

  - Users can configure per-user alert thresholds (default: 0.7 / 70%)
  - When a prediction's probability_yes exceeds the user's threshold,
    an alert is automatically created and stored in the database.
  - Alerts include: message, risk level, probability, timestamp.
  - Users can retrieve their alert history via GET /alerts.
  - Alert channels: in_app (default), email, sms (configurable).

    > uvicorn api.main:app --reload --port 8000

    > cd frontend
    > npm install
    > copy .env.example .env

    > npm run dev

  Step 7: Open http://localhost:3000 in your browser

  OPTION 2: DOCKER COMPOSE (Full Stack)
  ──────────────────────────────────────

    > docker compose up --build

    This starts all 3 services:
    - API:      http://localhost:8000
    - MLflow:   http://localhost:5000
    - Frontend: http://localhost:3000
